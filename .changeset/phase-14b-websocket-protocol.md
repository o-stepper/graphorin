Phase 14b — WebSocket protocol contract + reference TypeScript client + Server-Sent Events fallback. Two new packages join the framework on top of the Phase 14a REST surface; `@graphorin/server` is extended with the WebSocket / SSE transports, the `POST /v1/session/ws-ticket` route, and the per-server delivery-layer commentary-phase trace sanitization at the wire-emission boundary.

`@graphorin/protocol` ships:

- **Single source of truth for the wire format** of the `graphorin.protocol.v1` WebSocket subprotocol. Both `@graphorin/server` and `@graphorin/client` import their schemas from this package so the two implementations cannot drift.
- **Browser-friendly bundle.** ESM-only; zero Node-only dependencies; the only runtime dependency is `zod@^3.25.0`. Bundle size on disk is well under the 10 KB minified-gzipped budget the runtime spec calls out.
- **`SUBPROTOCOL_NAME` + `PROTOCOL_VERSION` + ticket helpers.** `formatTicketSubprotocol(value)` returns the `'ticket.<value>'` token browser clients attach to the `Sec-WebSocket-Protocol` header (because the WebSocket browser API forbids custom headers); `parseTicketSubprotocol(...)` and `negotiateSubprotocol(...)` are the matching server-side helpers.
- **`ClientMessage` discriminated union (Zod schema + types).** Six variants covering the JSON-RPC-shaped control plane: `initialize`, `subscription.subscribe`, `subscription.unsubscribe`, `run.cancel`, `ping`, plus the MCP-compatible `notifications/cancelled` notification. Strict object validation + `safeParse` round-trip on every frame.
- **`ServerMessage` discriminated union (Zod schema + types).** Nine variants: JSON-RPC `result` / `error` responses (`{ jsonrpc, id, result | error }`), typed push events (`{ kind: 'event', subject, type, payload, eventId }`), lifecycle frames (`{ kind: 'lifecycle', subscriptionId, status }`), async error frames (`{ kind: 'error', code, message }`), `pong`, `subscribed` / `unsubscribed`, and `replay-marker`. Per-variant type guards (`isEventFrame`, `isLifecycleFrame`, `isPongFrame`, `isReplayMarkerFrame`, `isRpcSuccess`, `isRpcFailure`, …) so consumers narrow the union without re-stringifying the discriminator.
- **Close-code taxonomy** (`CLOSE_CODE_VALUES`, `closeCodeFor`, `closeCodeReason`) covering the 4001 (`auth.required`) … 4008 (`protocol.violation`) range from the runtime spec, plus the JSON-RPC error-code catalogue (`RPC_ERROR_CODES.PARSE_ERROR` … `RUN_NOT_FOUND` / `SUBSCRIPTION_NOT_FOUND`).

`@graphorin/client` ships:

- **`GraphorinClient` ergonomic façade.** Constructor accepts `{ baseUrl, auth, transport?, reconnect?, WebSocket?, EventSource?, fetch?, clientId? }`; `transport: 'auto'` (default) tries WebSocket first and falls back to SSE when the WS handshake fails; `'ws'` and `'sse'` opt into a single transport.
- **Bearer + ticket auth flows.** Bearer-token strategy is used by Node SDKs / CLI clients (the `Authorization: Bearer <token>` header rides the WS upgrade through the 3-arg WebSocket constructor honoured by `ws` and most polyfills). Ticket strategy is the browser path: the supplied `ticketProvider()` is invoked at every connect, and the value is attached as a second `Sec-WebSocket-Protocol` token (`ticket.<value>`).
- **`subscribe({ target, id })`.** Returns a `Subscription` whose `events()` method yields a fully-typed `AsyncIterable<ServerEventFrame>` so consumers can `for await (const event of sub.events())`. Subscription targets cover every recognised subject shape: `'session'` / `<id>`, `'agent'` / `<id>` + `runId`, `'workflow'` / `<id>`, and `'run'` / `<runId>` (with optional `sessionId`).
- **Initialize handshake.** `connect()` automatically sends the `initialize` JSON-RPC request after the WebSocket is open and waits for the matching reply before resolving. Subsequent `subscribe()` / `cancel()` / `ping()` calls inherit the initialised connection.
- **`cancel(runId, opts)` + `cancelNotify(requestId)` + `resume(runId, directive, opts?)`.** `cancel` sends a `run.cancel` JSON-RPC request and resolves with the server's reply; `cancelNotify` emits the MCP-compatible `notifications/cancelled` notification (no `id`, no reply expected); `resume` POSTs to the Phase 14a REST endpoint `/v1/runs/:runId/resume` (resumes are durable + idempotent + body-carrying, which maps cleanly onto REST instead of the WebSocket control plane). The `resume` call honours an optional `idempotencyKey` and reuses the configured `fetch` implementation so SDK consumers can plug in their own networking stack.
- **Reconnect with replay buffer resume.** `computeBackoffMs(attempt, policy)` implements full-jitter exponential backoff (`AWS Architecture Blog` formula); on transport close the client resubscribes every active subscription with the recorded `lastEventId` so the server replays missed events from the per-subject replay buffer.
- **SSE fallback transport.** `openSseTransport(...)` opens an `EventSource` (browser-native or polyfill-injected via the `EventSource?` option). The SSE transport is read-only; control-plane operations on an SSE-only client raise a typed `TransportFailedError` so the consumer falls back to REST.
- **Typed error hierarchy.** `GraphorinClientError` base class + per-kind subclasses (`ClientNotConnectedError`, `TransportFailedError`, `SubprotocolMismatchError`, `AuthFailedError`, `ProtocolViolationError`, `SubscriptionNotFoundError`, `ClientAbortedError`, `InvalidServerFrameError`) so consumers pattern-match without `instanceof` quirks across module boundaries.
- **Browser bundle hygiene.** Zero Node-only dependencies; the only runtime dependencies are `@graphorin/protocol` and `zod`. The compiled `dist/` weighs ≈ 8 KB minified-gzipped — well under the 50 KB budget the runtime spec calls out.

`@graphorin/server` extensions:

- **`@graphorin/server/ws/` family.** Exposes `createWsDispatcher(...)`, `createWsTicketStore(...)`, `createReplayBuffer(...)`, `tryParseSubject(...)`, `requiredScopeFor(...)`, `isSubjectAllowed(...)`, and `createWsUpgradeEvents(...)` so operators can plug a custom Hono app on top of the same primitives.
- **In-memory ticket store.** Issues short-lived (default 5 min), single-use tickets bound to `(tokenId, scopes)` pairs. The `POST /v1/session/ws-ticket` route mints a ticket from the bearer-authenticated request; the WS upgrade handler consumes it transparently.
- **Per-subject replay buffer.** Default 1000 events per subject, 5 min TTL; configurable per-server via `server.stream.replayBuffer.{ maxEvents, ttlSeconds }`. On reconnect with `sinceEventId`, the dispatcher replays buffered events; on overflow it emits a `replay-marker` frame so the consumer is informed about the gap.
- **Strict subject grammar.** Recognises `session:<id>/events`, `session:<id>/runs/<runId>/events`, `agent:<id>/runs/<runId>/events`, `workflow:<id>/events`, `memory/conflicts`, `audit/events`. Wildcards (`session:*/events`) are rejected per the runtime spec and deferred to v0.2+. Per-subject scope check enforced at subscribe time using the `<resource>:<action>[:<id>]` grammar from `@graphorin/security/auth`.
- **WebSocket dispatcher.** Routes events to all matching subscriptions, validates every outbound frame against the `@graphorin/protocol` Zod schema before the bytes hit the wire, applies the delivery-layer commentary-phase trace sanitization to event payloads, and reports back through an optional `onWarn` sink for protocol violations / queue overflows / subject rejections.
- **Per-connection backpressure.** The dispatcher tracks an outstanding-event counter per subscriber and consults the optional `WsSubscriberHandle.bufferedAmount()` sample on every emit. When either the counter exceeds `server.stream.perConnectionQueueLimit` (default 1000) or the buffered byte sample exceeds `perConnectionQueueLimit * 1024`, the dispatcher closes the connection with the Graphorin `flow.throttled` close code (4006) and unregisters the subscriber so the client can reconnect against the replay buffer instead of accumulating an unbounded send queue. Send-throw failures also escalate to the same `flow.throttled` close so half-broken sockets do not back-pressure the whole event flow.
- **`POST /v1/session/ws-ticket` route.** HTTP-authenticated; requires the `agents:invoke` scope (or any wildcard cover); returns `{ ticket, expiresAt, ttlMs }` on success.
- **`@hono/node-ws` upgrade handler.** Wires the `WS /v1/ws` endpoint through the dispatcher: validates the subprotocol (rejects unknown variants with close code `4008 protocol.violation`), verifies the bearer-token / ticket auth inline (the HTTP auth middleware is intentionally bypassed for the upgrade path so the close-code semantics are preserved), and bridges the per-connection `WSContext` callbacks to the dispatcher's RPC handler. Hono's `WebSocketServer` (the underlying `ws` implementation) is configured with a `handleProtocols` policy that selects `graphorin.protocol.v1` so the negotiated subprotocol is echoed back to the client.
- **Server-Sent Events fallback.** `GET /v1/sessions/:id/events` streams events to non-WebSocket-friendly clients via the same Hono app. Uses the standard `Last-Event-ID` reconnect mechanism: when the client supplies the header on resume, the responder walks the dispatcher's replay buffer from that cursor and applies the same sanitization. Emits periodic comment-line heartbeats (default 15 s) so corporate proxies do not close the idle connection.
- **Delivery-layer commentary-phase trace sanitization.** New `@graphorin/server/commentary` module exports `createDeliveryCommentarySanitizer(...)` plus the `DEFAULT_DELIVERY_COMMENTARY_PATTERNS` catalogue (seven default patterns covering `tool.call.start` / `tool.call.delta` / `tool.call.end` / `tool.execute.end` / `agent.fanout.*` / `context.compacted` / `agent.model.fellback` event signatures). The sanitizer walks every string leaf of a `ServerEventFrame.payload` and applies the configured policy (`'wrap'` (default) wraps detections in `<<<commentary>>>...<<</commentary>>>`; `'strip'` removes them; `'pass-through'` is the operator opt-out for trusted deployments). Per-server scope (`server.ws.commentarySanitization.{policy, applyToEvents}` config); applied bytes-equal across the WebSocket dispatcher (`'ws'` transport label) and the SSE delivery boundary (`'sse'` label) so the second pass on a previously-sanitized payload is idempotent. Audit + counters surfaced through an optional `DeliveryCommentarySink.onDecision` hook so Phase 14c can wire `appendAudit` + Prometheus counters without modifying the sanitizer.
- **WebSocket / SSE config block.** New `server.ws.{ enabled, path, ticketTtlMs, commentarySanitization }`, `server.sse.{ enabled, path, keepAliveMs }`, and `server.stream.{ disconnectPolicy, disconnectGracePeriodMs, replayBuffer, perConnectionQueueLimit }` fields on `ServerConfigSpec`; every field has a documented production-ready default. `server.wsDispatcher` and `server.wsTickets` are exposed on the `GraphorinServer` handle so consumer code (and tests) can inspect / drive the WS surface programmatically.
